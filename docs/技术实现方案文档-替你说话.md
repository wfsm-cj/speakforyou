## 六、核心功能流程设计

### 6.1 用户认证（JWT）

```
1. 用户提交 {username, password}
2. AuthService 从 MySQL 查询用户 → BCrypt 比对密码
3. 验证通过 → 生成 JWT Token（payload: userId, username, role）
4. 返回 {token, userInfo} → 前端存入 localStorage
5. 后续所有请求在 Header 携带 Authorization: Bearer {token}
6. JwtAuthFilter 拦截请求 → 解析验证 Token → 设置 SecurityContext
7. 登出时将 token 加入 Redis 黑名单（TTL = token 剩余有效期）
```

Spring Security 配置要点：
- 关闭 CSRF（JWT 天然防护）
- 设置 SessionCreationPolicy.STATELESS
- 放行：登录接口、静态资源
- 其余接口均需认证

### 6.2 快速回复模式

#### 整体流程（SSE 推送）

```
1. 前端 POST /api/quick-reply/stream（Header 带 JWT）
   请求体：{message, personaId, sceneId}
   响应类型：text/event-stream（SSE）
2. Controller 创建 SseEmitter（超时 60s）并立即返回
3. 限流检查（RateLimitService）
   - 有自定义 Key → 跳过
   - 无 → Redis INCR rate_limit:{userId}:{date}，超过 30 次则拒绝
4. 查询人格（persona 表）和场景（scene 表）
5. RAG 检索：
   a. 将用户消息通过 EmbeddingModel 向量化
   b. 在 PgVectorEmbeddingStore 中检索 Top5 相似话术
   c. 可按 metadata.persona_type 过滤
6. 构造 Prompt（见下方 Prompt 设计）
7. 获取 StreamingChatModel：
   - 用户有自定义 Key → AiModelFactory 运行时创建模型实例
   - 无 → 使用系统默认 StreamingChatModel Bean
8. 调用 StreamingChatModel.chat()，注册回调：
   - onPartialResponse → emitter.send(token)
   - onCompleteResponse → emitter.complete()
   - onError → emitter.completeWithError()
9. 前端通过 fetch + ReadableStream 实时接收 token → 流式渲染
10. 流式完成后，前端解析完整 JSON 数组 → 渲染为 3 张回复卡片
```

#### Prompt 设计

```
[System Prompt]
你是一位职场沟通助手。你的人格设定如下：
- 人格风格：{persona.name}
- 风格描述：{persona.description}
- 说话语气：{persona.tone}

当前沟通场景：{scene.name} - {scene.description}
{scene.promptHint}

请根据以上人格和场景，针对对方发来的消息，生成3条不同的回复建议。

## 参考话术（来自知识库）
{ragResults — 5条检索到的相似话术}

## 输出要求
请严格按以下 JSON 数组格式输出3条回复，不要输出其他内容：
[
  {"id": 1, "reply": "回复内容1", "style": "风格特点"},
  {"id": 2, "reply": "回复内容2", "style": "风格特点"},
  {"id": 3, "reply": "回复内容3", "style": "风格特点"}
]

要求：
1. 3条回复要有明显差异性（正式程度、长短、语气）
2. 回复自然得体，符合职场沟通规范
3. 不要生成不当、攻击性或不专业的内容
4. 每条回复控制在 200 字以内

[User Message]
对方发来的消息："{userMessage}"
```

### 6.3 对话模式

#### 整体流程（WebSocket 推送）

```
1. 前端通过已建立的 WebSocket 连接发送：{sessionId, message}
2. ChatWebSocketHandler 接收 → 从 session attributes 获取 userId
3. 限流检查（同快速回复逻辑）
4. 加载对话上下文：
   a. 优先从 Redis（chat_memory:{sessionId}）读取
   b. 缓存未命中 → 从 MySQL chat_message 表按 sequence 加载
   c. 写入 Redis 缓存
5. 查询该会话绑定的人格 → 构造 System Prompt
6. 组装完整消息列表：System Prompt + 历史消息 + 当前用户消息
7. 获取 StreamingChatModel（同快速回复的 AiModelFactory 逻辑）
8. 调用 StreamingChatModel.chat()：
   - onPartialResponse → session.sendMessage(token JSON)
   - onCompleteResponse → session.sendMessage(complete JSON)
   - onError → session.sendMessage(error JSON)
9. 流式完成后：
   a. 将用户消息 + AI 完整回复存入 MySQL chat_message 表
   b. 更新 Redis 缓存（追加新消息，超过 50 条则裁剪最早的）
   c. AI 的回复作为"你的回复"进入上下文
```

#### 上下文记忆管理

自定义 `PersistentChatMemoryStore` 实现 LangChain4j `ChatMemoryStore` 接口：
- `getMessages(sessionId)`：Redis 优先 → MySQL 兜底 → 回填 Redis
- `updateMessages(sessionId, messages)`：双写 MySQL + Redis
- 配合 `MessageWindowChatMemory`（maxMessages=50）实现窗口裁剪

根据用户传入的不同参数（会话绑定的人格），动态构建不同的 AiService 实例。

#### Prompt 设计

```
[System Prompt]
你是用户的职场沟通顾问。你的人格设定如下：
- 人格风格：{persona.name}
- 风格描述：{persona.description}
- 说话语气：{persona.tone}

## 你的工作方式
1. 用户会粘贴别人发来的消息给你
2. 你需要根据你的人格风格，给出最合适的回复建议
3. 你生成的回复将直接被用户复制使用，所以只输出回复内容本身
4. 不要输出分析过程、解释或多余的话
5. 回复要自然得体，符合职场沟通规范
6. 不要生成不当、攻击性或不专业的内容
```

### 6.4 RAG 话术知识库

#### 流程

```
[数据导入阶段]
1. 读取 speech_samples.json（250条话术，5种人格 × 50条）
2. 逐条通过 EmbeddingModel（text-embedding-v3）向量化
3. 附加 metadata（persona_type、scene_type）
4. 存入 PgVectorEmbeddingStore

[检索阶段 — 仅快速回复模式使用]
1. 用户消息 → EmbeddingModel 向量化
2. PgVectorEmbeddingStore 相似度检索（Top5，minScore=0.6）
3. 可按 metadata.persona_type 过滤，匹配当前人格
4. 检索结果注入到快速回复 Prompt 的"参考话术"部分
```

LangChain4jConfig 中需手动创建三个 Bean：
- `EmbeddingModel`：读取 `app.embedding` 配置，使用 OpenAiEmbeddingModel 指向 DashScope
- `PgVectorEmbeddingStore`：读取 `app.pgvector` 配置
- `EmbeddingStoreContentRetriever`：组合上面两个 Bean，设置 maxResults=5、minScore=0.6

#### 话术数据初始化

- 数据文件：`src/main/resources/data/speech_samples.json`
- 格式：`[{persona_type, scene_type, text}, ...]`
- 导入方式：编写测试类，通过 `EmbeddingModel` 向量化后存入 `PgVectorEmbeddingStore`
- 每种人格 50 条样本，覆盖 4 种场景

### 6.6 限流服务（RateLimitService）

```
每次 AI 请求前调用：
  1. IF 用户有自定义 API Key → 直接放行
  2. ELSE:
     a. Redis INCR rate_limit:{userId}:{今天日期}
     b. 若为首次请求（count=1），设 TTL 到当天 23:59:59
     c. IF count > 30 → 拒绝请求，返回"今日次数已用完"提示
     d. ELSE → 放行
```

### 6.7 流式通信设计

#### 6.7.1 快速回复模式 — SSE

```
通信方式：HTTP POST → SseEmitter（单次请求-响应，用完即关）
认证方式：请求 Header 中的 Authorization: Bearer {JWT}（Spring Security 统一拦截）
端点：POST /api/quick-reply/stream（produces: text/event-stream）

前端接收：
  fetch() + response.body.getReader()（ReadableStream）
  逐 chunk 读取 → 实时渲染 → 流结束后解析完整 JSON
```

#### 6.7.2 对话模式 — WebSocket

**连接管理**：
```
1. 前端建立 WebSocket 连接：ws://host/ws/chat?token={JWT}
2. 服务端 HandshakeInterceptor 拦截握手：
   a. 从 URL 参数提取 JWT
   b. 验证 Token 有效性
   c. 通过 → userId 存入 WebSocketSession attributes
   d. 失败 → 拒绝握手
```

**消息协议（JSON）**：

客户端 → 服务端：
```json
{"sessionId": 100, "message": "对方发来的消息"}
```

服务端 → 客户端：
```json
{"type": "token", "data": "部分文本"}
{"type": "complete", "data": "完整回复内容"}
{"type": "error", "data": "错误信息"}
```

**处理流程**：
```
ChatWebSocketHandler.handleTextMessage():
  1. 解析客户端 JSON → 获取 sessionId、message
  2. 从 session attributes 获取 userId
  3. 调用 ChatService.streamChat(userId, sessionId, message, wsSession)
  4. Service 内部调用 StreamingChatModel.chat()，回调中：
     - onPartialResponse → wsSession.sendMessage(token JSON)
     - onCompleteResponse → wsSession.sendMessage(complete JSON)
     - onError → wsSession.sendMessage(error JSON)
```

## 十、安全与防护

| 防护点 | 实现方式 |
|--------|---------|
| 用户认证 | JWT + Spring Security |
| 密码存储 | BCrypt 加密 |
| API Key 存储 | MVP 明文存储 |
| CSRF | JWT 无状态天然防护 |
| 接口限流 | Redis 计数器 30次/天 |
| AI 内容安全 | Prompt 约束 + 通义千问内容审核 |
| 聊天内容 | 快速回复模式不持久化用户粘贴内容 |
| SSE 认证 | HTTP Header Bearer Token（Spring Security 统一拦截） |
| WebSocket 认证 | 握手阶段验证 JWT |

## 十一、风险与应对

| 风险 | 应对措施 |
|------|---------|
| AI 生成不当内容 | Prompt 约束 + 通义千问内容审核 |
| 系统 Key 滥用 | 限流 30次/天 + 必须登录 |
| SSE 超时中断 | SseEmitter 设置 60s 超时 + 前端错误重试 |
| WebSocket 连接中断 | 前端自动重连机制 |
| LangChain4j beta 不稳定 | 锁定 1.11.0-beta19 |
| 通义千问 API 限流 | 捕获 429 错误，提示稍后重试 |
| 用户自定义 Key 无效 | 设置时发送测试请求验证 |
